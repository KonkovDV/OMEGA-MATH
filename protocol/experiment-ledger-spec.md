# OMEGA Experiment Ledger Specification

Every research run that produces artifacts, computations, or formal-proof attempts must be
recorded in a structured experiment ledger.

## Purpose

1. Trace which problems were worked on, when, and with what results.
2. Provide a reproducibility surface: any logged run can be re-executed from its parameters.
3. Feed the Planner agent with prior-run context when scheduling the next research cycle.

## Ledger Location

Per-problem ledgers live inside the problem workspace:

```
research/active/<problem-id>/experiments/ledger.yaml
```

A global summary index across all active problems:

```
research/active/experiment-index.yaml
```

## Run Record Schema

Each entry in `ledger.yaml` is a run record:

```yaml
- run_id: "<problem-id>-<YYYYMMDD>-<seq>"
  problem_id: "<registry problem ID>"
  started: "<ISO 8601 timestamp>"
  finished: "<ISO 8601 timestamp or null>"
  route: "experiment-first | proof-first | survey-first"
  agent: "<agent role that executed>"
  status: "running | completed | failed | abandoned"
  verdict: "positive | negative | inconclusive | null"
  parameters:
    description: "<what was tried>"
    bound: "<search bound, instance size, or proof depth>"
    model: "<LLM or prover model used, if any>"
    hardware: "<CPU/GPU note when relevant>"
  artifacts:
    - path: "<relative path to output file>"
      type: "log | dataset | proof-draft | prover-result | counterexample | plot"
      checksum: "<SHA-256 of the artifact, optional>"
  parent_run: "<run_id of the predecessor, or null>"
  notes: "<free-text operator notes>"
```

## Required Fields

- `run_id`, `problem_id`, `started`, `route`, `agent`, `status`

All other fields are recommended but may be null for incomplete or abandoned runs.

## Lifecycle

1. **Open**: Experimentalist or Prover creates a new entry with `status: running`.
2. **Close**: Agent updates `status`, `finished`, `verdict`, and `artifacts` when the run ends.
3. **Abandon**: If a run is interrupted, set `status: abandoned` with a note explaining why.
4. **Chain**: A follow-up run references `parent_run` to build a traceable lineage.

## Verdict Semantics

| Verdict | Meaning |
|---------|---------|
| `positive` | Found a counterexample, improved a bound, or produced a proof draft that passes internal review |
| `negative` | Exhaustive search within bounds found nothing; formal verifier rejected all candidates |
| `inconclusive` | Partial progress; result is useful context but does not resolve the sub-question |

## Integration Points

- **Planner**: reads the ledger before scheduling the next run to avoid repeating failed approaches.
- **Writer**: extracts evidence class and artifact references from the ledger when drafting a note.
- **Reviewer**: cross-checks claims against ledger entries; a claim without a ledger run is flagged.
- **Verification Pipeline**: `verification-pipeline.md` Stage 1 (Artifact check) confirms that
  every artifact listed in the ledger entry actually exists on disk.
- **Proof-first runs**: the ledger records campaign execution, but verifier-visible claim status belongs in `artifacts/prover-results/<run-id>.yaml` per `protocol/prover-result-contract.md`.

## Global Index

`experiment-index.yaml` aggregates the latest run plus the current workflow snapshot per active problem:

```yaml
- problem_id: "<ID>"
  latest_run: "<run_id>"
  latest_verdict: "<verdict>"
  total_runs: <N>
  active_route: "experiment-first | proof-first | survey-first | null"
  current_stage: "brief | novelty | triage | plan | experiment | prove | survey | results | paper | referee | closed | null"
  current_owner: "<planner | librarian | analyst | experimentalist | prover | writer | reviewer | null>"
  workflow_status: "ready | running | blocked | completed | null"
  active_run_id: "<run_id or null>"
  blocked: true | false
  last_updated: "<ISO 8601>"
```

This index is regenerated by `scripts/generate-experiment-index.py` and is refreshed automatically by `scripts/omega-runner.py start|finish|proof-result`.

## Conventions

- Ledger files are append-only during a session. Edits to past entries must add a `corrected_at` field.
- Artifact paths are relative to the problem workspace root.
- Checksums are optional but recommended for any file that supports a publication-grade claim.
- Do not store large binary artifacts in git; use `.gitignore` and record the checksum only.
- A `positive` ledger verdict does not by itself mean a theorem is proved; use the paired prover result artifact for claim status.
- Workflow state and ledger state are complementary: the ledger records execution history, while the workflow snapshot records where the investigation currently sits and who should own the next step.
